Q1: FoundationDB separates these two systems so they can be scaled independently based on what the database needs at the time. The Transaction Management System handles the in-memory work which is fast. The Storage System just focuses on keeping the actual data safe on disk and answering read queries.
Q2: The Resolvers inside the Transaction Management System are responsible for checking read-write conflicts between transactions.This check only happens right at the end during the commit phase. If a conflict is detected then, the transaction just fails and has to retry.
Q3: The read version makes sure that the transaction sees a correct and consistent snapshot of the database from a specific point in time. It prevents the transaction from accidentally reading incorrect updates from other concurrent transactions. Showing how FDB guarantees consictency  strict serializability for its users.
Q4: Snapshot reads give you a consistent view of the data, but they aren't checked for conflicts during the commit phase. So a snapshot read will never cause your transaction to abort if someone else modifies the data. They can be used for running large analytics queries but you aren't writing anything back that depends on it.
Q5: Recovery is fast because FDB doesn't have to pause the system to replay from disk. During normal operations, LogServers save the recent transaction logs and stream them to the StorageServers. If the system crashes, a new transaction layer starts, and the StorageServers just pull log data they missed from the old LogServers not blocking new queries.
